# 步骤3评估指南

## 评估标尺

| 复杂度 | 特征 | 参考工时(L2单点) |
|--------|------|-----------------|
| 低 | 纯CRUD / 正则匹配 / 固定规则≤5条 / 标准库直调 | 0.5 - 1 人天 |
| 中 | 规则>5条有条件分支 / 简单LLM Prompt / 前端有状态管理 | 1.5 - 3 人天 |
| 高 | 复杂计算逻辑 / LLM推理需多轮调优 / 前端重交互 / 涉及CV模型训练 | 3 - 8 人天 |
| ⚠ | 受前置依赖阻塞，当前无法评估 | 先标注，追踪依赖解除 |

## L1→L2 拆解格式

每个稳态工具按4个L1模块拆解：

```
### L1: 数据接入
| L2 功能点 | 复杂度 | 人天 | 业务专家备注 |
|-----------|--------|------|-------------|

### L1: 业务流程编排（核心）
| L2 功能点 | 复杂度 | 人天 | 业务专家备注 |

### L1: 用户前端
| L2 功能点 | 复杂度 | 人天 | 备注 |

### L1: 管理后台
| L2 功能点 | 复杂度 | 人天 | 备注 |

**合计: XX 人天**
```

### 业务备注要求

每个L2功能点必须标注具体的业务规则数量级，不能只写"数据清洗"。示例：
- ✅ "约15-20条清洗规则：集团合并/税务/管理口径各有不同的科目映射和抵消规则"
- ❌ "数据清洗功能"

遇到下列情况时，切换到**业务专家**角色补充：
- 财务: 报表模板种类、口径套数、会计科目数、审核规则条数
- 质量: 质检标准、缺陷分类数、检测项清单
- 法务: 风险条款分类数、标准条款库规模
- CV: 检测项数量、训练样本量要求
- LLM: Prompt分类数、调优迭代轮数预期

## 共用组件识别

标识方法：
1. 横向对比所有工具的L2功能点
2. 出现≥2次且实现逻辑相同的点 → 提取为共用组件
3. 共用组件单独评估人天，各工具的L2备注中标注 `[共用Cx]`
4. 输出交叉引用矩阵（✓标注）

常见共用组件类别：
- 文件解析服务 (C1): Excel/CSV/PDF/DOCX
- 报表模板引擎 (C2): 模板DSL/填充/图表
- 审批流引擎 (C3): 流程/节点/API/日志
- 文件导出服务 (C4): PDF/XLSX/DOCX
- 操作日志审计 (C5): 中间件/查询/面板
- 定时任务调度 (C6): Cron/监控/管理
- 规则引擎 (C7): DSL/执行/管理
- LLM调用网关 (C8): 统一API/Prompt管理/Token监控
- 工业视觉管道 (C9): 相机SDK/采集/推理/归档
- 可视化图表库 (C10): 图表组件/看板布局
- 文件存储服务 (C11): 上传/预览/版本

## 敏态层评估

数字员工引擎拆解为：
1. 共享引擎 (3Agent共用): 意图理解/流程规划/Skill加载/工具调度/结果校验/复盘优化/上下文管理/Harness框架 → 约38人天
2. Per-agent Harness定制: 每个Agent独立配置领域Prompt+少样本示例+话术 → 约3-5人天/Agent

## Skill管理面评估

独立于工具侧的Skill开放体系管理模块：
- Skill发现中心: 语义搜索+标签过滤 → 约4人天
- Skill权限鉴权: RBAC+Token+配额 → 约6人天
- Skill版本管理(V1简化): SemVer+切换+回滚 → 约3人天
- Skill监控计量(V1简化): 调用统计+成功率+延迟 → 约3人天
- CLI/API调度网关: 入口+路由+适配+异步 → 约5人天

## L1/L3评估

- L1 WEB工作台: 货架式首页+SSO+搜索+入口切换+框架 → 约12人天
- L1 IM集成: 钉钉+企微开发+任务中心+结果查看 → 约12人天
- L3 RAG知识库: 文档解析+向量化+检索+管理界面 → 约10人天
- L3 日志监控: 全链路trace+监控大盘+告警 → 约5人天
- L3 推理存储: 模型适配+LDAP+DB部署 → 约9人天

## Excel输出规范

- 使用openpyxl生成
- 颜色标注: 低(绿色#C6EFCE) / 中(黄色#FFEB9C) / 高(红色#FFC7CE)
- L1模块行使用浅灰蓝(#F2F2F2)区分
- 小计/合计行使用浅黄(#FFF2CC)
- 工具间用粗体区分行分隔
- 表头: 深蓝(#2F5496) + 白色字体

Sheet结构:

2. 14工具明细: 完整L1→L2拆解
3. 敏态引擎: 共享引擎+Per-agent
4. Skill管理: 5个管理模块
5. L1交互层: WEB+IM+框架

## V1/V2边界方法论

V1/V2边界分析在以下条件触发：
- 项目有明确的"先通后优"、"MVP优先"或分期交付策略
- 存在明确的技术约束（如V1不做系统对接）

产出：
- 功能模块矩阵: 每个模块标注 V1范围 vs V2预留
- 前置依赖阻塞项: V1功能中受外部依赖阻塞的 → 标注 "◆" 并追责提供方
- V2预留功能不评估人天
